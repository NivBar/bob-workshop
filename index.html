What this workshop is about

Before any system is changed, improved, or extended — someone needs to truly understand it. Most organisations skip this. Teams make changes to systems they only partially understand. Things break. Audits fail. Costs spiral.

This workshop shows a different approach — using IBM Bob to read, analyse, and document a real software project before touching a single line of code. Every step produces a real document. By the end, you will have the same documentation package that a professional team would spend weeks producing — created directly from the code.

We use the OpenMRS Appointment Scheduling Module — a real, open-source healthcare project — as our practice system.
Repository: github.com/openmrs/openmrs-module-appointmentscheduling

Clone the Repository

Download the project so Bob can read and analyse it locally

Bob works with code that is on your computer — it reads the actual files. Before Bob can analyse anything, the project needs to exist locally. "Cloning" is simply downloading a full copy of the project from GitHub.

Before you start, confirm you have:

  • Bob is installed and you are signed in with your IBMid
    Don't have Bob? Download at bob.ibm.com/download
  • Git is installed — open a terminal and run git --version to confirm
  • A stable internet connection — the repository is ~50 MB
Option A — Terminal
Run this command
git clone https://github.com/openmrs/openmrs-module-appointmentscheduling.git
Option B — GitHub Desktop
  1. Open GitHub Desktop
  2. Click File → Clone repository
  3. Click the URL tab and paste the repository URL
  4. Choose where to save it, then click Clone
Then open in Bob
  1. Open Bob → File → Open Folder
  2. Navigate to the cloned folder and click Open
  3. Wait 20–30 seconds for Bob to index the files
Verification — paste into Bob to confirm files are loaded
How many Java files are in this project?

If Bob gives you a specific number, the files are loaded and you are ready to continue.

1

Understand the System

Ask Bob to read the entire project and explain it in plain English

Most organisations have systems that only one or two people truly understand. When those people leave, the knowledge walks out with them. Bob reverse-engineers that knowledge from the code itself — in under a minute.
📋 Prompt 1 — The Executive Briefing
I have just been given responsibility for this codebase and I know nothing about it. Read the entire project and give me an executive briefing in plain English: 1. What does this system do? 2. Who uses it and why does it matter to them? 3. What business problem does it solve? 4. How big and complex is it — roughly? Write your answer as if explaining to a hospital administrator who has never seen code. No technical jargon. Plain English only. Save your response as openmrs-workshop/01-EXECUTIVE-BRIEFING.md
What to expect
  • Bob reads every file in 20–40 seconds
  • Identifies this as a healthcare scheduling system
  • Mentions patients, doctors, clinic slots, workflows
  • Gives a rough sense of scale
What good looks like
  • A non-technical person can read it immediately
  • System purpose is clear and specific
  • Number of files and modules mentioned
If I had inherited this system yesterday, how long would it have taken me to reach this level of understanding — without Bob? A day? A week?
2

Map the Structure

Understand how the system is organised internally — not just what it does

Understanding structure is what allows you to make changes safely. If you don't know which part of a system does what, any change you make is a guess. Think of this like a floor plan — you can't safely renovate without one.
📋 Prompt 2 — The Architecture Map
Explain the architecture of this system to me. 1. What are the main structural layers — what is each one responsible for? 2. What are the key components and how do they connect? 3. Draw me a simple text diagram showing the overall structure. Explain everything in plain English. Use no Java terminology I won't understand. If you must use a technical term, explain it in brackets immediately after. Save your response as openmrs-workshop/02-ARCHITECTURE-MAP.md
What to expect
  • A text diagram of the system structure
  • Web layer, service layer, data layer identified
  • Two modules (api and omod) explained
What good looks like
  • Diagram is visually clear at a glance
  • Each layer explained by what it does — not how
  • No unexplained jargon remains
Does my own organisation have a document like this for its critical systems? If a new person joined tomorrow, could they get this level of understanding in under a minute?
3

Find the Risks

Surface hidden problems — technical debt, outdated tech, incomplete work

Every organisation has systems that carry hidden risk — technical debt that has never been catalogued, security gaps that have never been formally identified. Bob surfaces those risks automatically, in plain language, so decision-makers can act on them.
📋 Prompt 3 — The Risk Assessment
Review this entire codebase and give me a risk assessment. 1. What are the biggest technical risks — things that could cause problems in production? 2. What is unfinished or incomplete — work that was started but never completed? 3. Are there any security concerns — places where data could be exposed or misused? 4. Is the technology up to date, or are there components that are outdated or unsupported? Present your findings as a prioritised list — most important first. Explain each finding in plain English, including what the business impact would be if it is not addressed. Save your response as openmrs-workshop/03-RISK-ASSESSMENT.md

Bob should find

Java 1.6 compiler target — end of life since 2006 Build pipeline running on Java 8 — also end of life TODO in AppointmentRequest.java — never completed TODO in PatientAppointment.java — never completed Core module importing a web component it should not No vulnerability scanning in the CI/CD pipeline
What to expect
  • Prioritised risk list, most critical first
  • TODO markers in source files flagged
  • Structural dependency violation identified
What good looks like
  • Business impact stated for every finding
  • A non-technical manager can prioritise from this
  • Ranked from most to least serious
How long would it take a consultant to produce this risk assessment manually — and what would it cost? What risks might exist in my own organisation's systems that nobody has ever formally identified?
4

Document Design Decisions

Record why the system was built the way it was — before someone forgets

Every software system is built on decisions that were almost never written down. Future teams make changes that inadvertently break assumptions — because they didn't know the assumptions existed. Decision Records protect future teams from repeating past mistakes.
📋 Prompt 4 — Capturing Key Decisions
Read this codebase and identify the 3 most significant decisions that were made when it was built — decisions about how it is structured, what tools it uses, and why. For each decision, write a short Decision Record with these sections: - Decision: what was decided (one sentence) - Context: why this decision needed to be made - Reasoning: why this option was chosen over alternatives - Consequences: what this decision means for anyone who wants to change the system in future Write each record in plain English. No technical jargon without explanation. Format each one clearly under a numbered heading. Save your response as openmrs-workshop/04-DESIGN-DECISIONS.md
What to expect
  • 3 structured Decision Records
  • Two-module split (api / omod) explained
  • Liquibase (database tracking tool) documented
  • Spring MVC (web framework) choice captured
What good looks like
  • Plain language — anyone can read it
  • "Consequences" section tells future teams what to respect
  • A manager could read this and understand the system
5

Assess Security

Conduct a structured security review — clear enough for non-technical decision-makers

Security reviews are expensive. A professional security consultant charges thousands per day. Bob conducts a structured security review in minutes — producing findings clear enough for non-technical decision-makers to act on.
📋 Prompt 5A — Focused Security Review
Conduct a security-focused review of this codebase. Look specifically for: 1. Places where user input is accepted but not properly validated or sanitised (sanitised means: checked to make sure it cannot be used to attack the system) 2. Any places where one part of the system is importing or using something it should not have access to 3. Any incomplete security features — work that was started but not finished 4. Any missing security checks in the automated build process For each finding: - Give it a severity rating: Critical, High, Medium, or Low - Explain what the problem is in plain English - Explain what could go wrong if it is not fixed - Suggest what should be done about it Save your response as openmrs-workshop/05-SECURITY-REVIEW.md

Expected findings

AppointmentRequest.java imports org.openmrs.web.WebUtil — structural violation Incomplete feature at AppointmentRequest.java line 42 Only the notes field has HTML sanitisation — other fields do not No OWASP vulnerability scan in .github/workflows/maven.yml
Also run
📋 Prompt 5B — Performance Assessment
Based on what you can read in this codebase, tell me how well this system is built to perform under real-world conditions. I want to understand: - How fast does it appear to be designed to respond? - How many users does it seem designed to handle at once? - How resilient is it — what happens if something goes wrong? - How easy is it to maintain and change — or is it fragile? - Where does the code fall short of what a production system should be? Derive your answer entirely from what the code tells you — not from assumptions. Write your findings as a structured list. Flag clearly anywhere the system is not meeting the standards it should. Save your response as openmrs-workshop/05B-PERFORMANCE-REVIEW.md
What to expect
  • No timeout or connection pool config found
  • No caching layer identified
  • Absence of explicit performance targets flagged as a gap
What good looks like
  • Bob is honest about what the code doesn't say
  • Gaps are flagged as gaps — not glossed over
  • Output is actionable for a compliance or audit team
6

Plan the First Safe Change

Having understood, mapped, and documented the system — now we can plan

This is the payoff. Every step before this was preparation. Teams that skip steps 1–5 and go straight to making changes are the ones who introduce bugs, break integrations, and create security vulnerabilities. The documentation produced in this workshop is the foundation that makes confident, safe change possible.
📋 Prompt 6A — Plan a New Feature
I want to add a patient waitlist feature to this appointment scheduling system. The feature should work as follows: - When all available time slots for a given appointment type are fully booked, patients should be able to join a waitlist - When a cancellation occurs, the first patient on the waitlist should be automatically offered the freed slot Using everything you know about this codebase from our earlier analysis: 1. What would need to be built — list the new components required 2. What existing parts of the system would be affected 3. What changes would need to be made to the database (the permanent store of information) 4. What are the risks — what could go wrong if this is built incorrectly? 5. What would need to be tested to confirm the feature works correctly? Base your plan entirely on the actual structure of this codebase. Reference real component names where relevant. Write the plan clearly enough for a project manager to use it to scope the work. Save your response as openmrs-workshop/06-FEATURE-PLAN-WAITLIST.md
Then evaluate the approach
📋 Prompt 6B — Evaluate Two Approaches
For the patient waitlist feature, I am considering two different approaches: Option A: Create a brand new section in the database specifically for the waitlist, linked to the existing appointment data. Option B: Use a component that already exists in this system called AppointmentRequest, and add a new status called "WAITLISTED" to it. Evaluate both options against this specific codebase. - What are the advantages and disadvantages of each? - Which do you recommend and why? - Are there any risks in the existing code I should know about before choosing? - Be honest — if neither option is ideal, say so and explain why. Append your response to openmrs-workshop/06-FEATURE-PLAN-WAITLIST.md under a new heading "## Approach Evaluation"
What to expect
  • Structured comparison of both options
  • Real component names referenced from the code
  • Clear recommendation with honest reasoning
What good looks like
  • Decision-maker can approve a technical direction with confidence
  • Tradeoffs acknowledged — not glossed over
  • A PM can use this to estimate and schedule work
